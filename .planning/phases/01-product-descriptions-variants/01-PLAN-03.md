<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API response → React component | `product.variants` data flows from GET /api/products/:id into the client component as a prop. Trust boundary: server-provided data, rendered client-side by React. |
| Customer click → handleAddToCart | User interaction triggers cart mutation. No authentication required — cart is local Zustand state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-01 | Tampering | OOS button click — disabled attribute | mitigate | `disabled={oos}` attribute prevents DOM-level click. `onClick={() => !oos && setSelectedSize(v.size)}` is a redundant JS guard. Both are present. Customer cannot select an OOS size through normal browser interaction. |
| T-01-03-02 | Tampering | Adding OOS variant to cart via JS console | accept | Frontend OOS check is a UX guard, not a security control. Stock enforcement must happen server-side at order creation (out of scope for Phase 01). Accepted as known limitation, tracked for Phase 03. |
| T-01-03-03 | Information Disclosure | `v.stock` value exposed in rendered HTML | accept | Stock counts are visible in the browser (positive stock = selectable, zero = OOS button). The exact stock number is not displayed — only the boolean OOS state. Standard e-commerce practice. |
| T-01-03-04 | Spoofing | Dead import `@/lib/cart` (getOrCreateCart) removal | mitigate | Removed in this plan. Eliminates the dead Medusa API call path that could reach an undefined/misconfigured endpoint. |
| T-01-03-05 | Denial of Service | `product.variants?.every(...)` on large variant arrays | accept | Variant count is bounded by the 7-item size pool (D-04). `.every()` on 7 items is O(7) — no performance concern. |
</threat_model>
