## CRITICAL BUGS (Exploitable or data-destroying)

---

### BUG-001: OTP Brute-Force — No Attempt Limit
**Severity**: CRITICAL  
**CWE**: CWE-307 (Improper Restriction of Excessive Authentication Attempts)  
**File**: `backend/src/api/store/auth/verify-otp/route.ts`

**Description**:  
The OTP verification endpoint has no limit on the number of incorrect attempts. An attacker who intercepts or guesses a victim's phone number can try all 1,000,000 six-digit combinations. The only protection is a 60-second rate limit on `/send-otp`, not on `/verify-otp`. An attacker can send one OTP, then make unlimited verification attempts in the 10-minute TTL window.

**Reproduction**:
```bash
# Send one OTP
POST /store/auth/send-otp { "identifier": "+919876543210" }

# Brute-force verify (no rate limit on this endpoint)
for otp in $(seq -w 100000 999999); do
  POST /store/auth/verify-otp { "identifier": "+919876543210", "otp": "$otp" }
done
```

**Fix**:
- Track attempt count per OTP in Redis (e.g., `otp_attempts:{identifier}`)
- Invalidate OTP after 5 failed attempts
- Add per-IP rate limiting (10 req/minute on verify endpoint)
- Implement exponential backoff

### BUG-002: Hardcoded Secrets in Version-Controlled Template
**Severity**: CRITICAL  
**CWE**: CWE-798 (Use of Hard-coded Credentials)  
**File**: `backend/.env.template` (lines 5–6)

**Description**:  
```
JWT_SECRET=supersecret
COOKIE_SECRET=supersecret
```
These are committed to version control. Any developer who copies `.env.template` to `.env` without changing values runs production with known secrets. If the `.env` file is ever accidentally pushed, or if the template is deployed as-is, sessions can be forged.

**Fix**:
- Replace with `JWT_SECRET=CHANGE_ME_GENERATE_WITH_openssl_rand_base64_32`
- Add `.env` validation at startup — fail fast if default values are detected
- Rotate these secrets for any environment where they have been used

### BUG-003: Payment Signature Replay Attack
**Severity**: CRITICAL  
**CWE**: CWE-347 (Improper Verification of Cryptographic Signature)  
**File**: `backend/src/api/store/checkout/verify-payment/route.ts`

**Description**:  
The `/verify-payment` endpoint validates the HMAC signature correctly, but does not check whether the same `(razorpay_order_id, razorpay_payment_id, razorpay_signature)` tuple has already been used. An attacker who captures a legitimate payment verification request can replay it to trigger duplicate order creation.

**Fix**:
- Store verified `payment_id` values in Redis or database with a unique constraint
- On verification: check if `payment_id` already exists; return 409 if so
- Set TTL on the Redis record proportional to expected order lifecycle

### BUG-004: Duplicate Orders on Network Retry
**Severity**: CRITICAL  
**CWE**: CWE-362 (Race Condition)  
**File**: `backend/src/api/store/checkout/complete/route.ts`

**Description**:  
`/store/checkout/complete` has no idempotency protection. If the frontend retries on a network timeout (or the user double-clicks), two identical requests will both succeed and create two separate orders in Medusa, potentially double-charging the customer.

**Fix**:
- Accept an `idempotency_key` header (e.g., UUID generated by frontend)
- Cache completed order results keyed by `idempotency_key` in Redis (TTL: 24 hours)
- On duplicate request with same key, return the cached response instead of re-running

### BUG-005: Email OTP Delivery Completely Missing
**Severity**: HIGH  
**File**: `backend/src/api/store/auth/send-otp/route.ts` (lines 51–56)

**Description**:  
When `identifier` is an email address, the OTP is logged to console in development and silently dropped in production. Users with email identifiers cannot authenticate.

```typescript
// Current code (line 51-56):
if (process.env.NODE_ENV === "development") {
  console.log(`[DEV] Email OTP for ${identifier}: ${otp}`)
} else {
  // TODO: Send email via Resend
}
```

**Fix**:
- Install `resend` npm package
- Implement `sendEmailOtp(email, otp)` function
- Replace the stub with actual delivery; throw on failure

### BUG-006: Checkout Complete — No Error Boundary, Generic 500
**Severity**: HIGH  
**File**: `backend/src/api/store/checkout/complete/route.ts` (line 78–82)

**Description**:  
The catch block returns the raw error message to the client:
```typescript
catch (error) {
  return MedusaResponse.json({ message: error.message }, { status: 500 })
}
```
Internal Medusa errors (database constraint messages, workflow errors) are exposed directly. This leaks schema information, internal IDs, and stack trace fragments.

**Fix**:
- Log full error server-side with request context
- Return sanitized message: `{ message: "Order could not be completed. Please try again." }`
- Map known error types to appropriate HTTP codes (400 for validation, 409 for conflict)

### BUG-007: MSG91 Credentials Exposed in Error Logs
**Severity**: HIGH  
**CWE**: CWE-532 (Insertion of Sensitive Information into Log File)  
**File**: `backend/src/api/store/auth/send-otp/route.ts`

**Description**:  
On MSG91 API failure, the error thrown may include the raw HTTP response body, which can contain auth tokens, template IDs, or diagnostic information from MSG91.

**Fix**:
- Catch MSG91 errors and log only: status code, sanitized message
- Never log `MSG91_AUTH_KEY` or response bodies that may contain it

### BUG-008: Fallback Email Generation Exposes Phone Number
**Severity**: HIGH  
**CWE**: CWE-330 (Use of Insufficiently Random Values)  
**File**: `backend/src/api/store/auth/verify-otp/route.ts` (line 37)

**Description**:  
When a customer authenticates with a phone number but no email is provided, a fabricated email is generated:
```typescript
const fallbackEmail = `${identifier.replace(/[^0-9]/g, "")}@poshakh.in`
// Result: 919876543210@poshakh.in
```
This encodes the customer's phone number in their email address, which is visible in Medusa admin, analytics, logs, and potentially customer-facing contexts. It also looks like spam to email providers if ever used for sending.

**Fix**:
- Generate email as `cust_{uuid}@noreply.poshakh.in`
- Store phone as the canonical identifier; email is an internal artifact only

### BUG-009: Shipping Hardcoded to India Only
**Severity**: HIGH  
**File**: `backend/src/api/store/checkout/complete/route.ts` (line 34)

**Description**:
```typescript
country_code: "in"  // hardcoded
```
Any customer attempting checkout from a different country will get this silently overridden. If international shipping is ever supported, this will incorrectly apply Indian rates and validation.

**Fix**:
- Accept `country_code` from the request's `shipping_address` object
- Validate it against allowed countries for the active region

### BUG-010: No Admin Route Authentication
**Severity**: HIGH  
**CWE**: CWE-306 (Missing Authentication for Critical Function)  
**File**: `backend/src/api/admin/custom/route.ts`

**Description**:  
```typescript
export const GET = async (req, res) => {
  return res.json({})  // No auth check whatsoever
```

- Apply Medusa's `authenticate` middleware to all admin routes
- Use `@medusajs/framework/http` middleware guards

## MEDIUM BUGS (Degraded behavior, non-exploitable data issues)

---

### BUG-011: OTP Stored Unencrypted in Redis
**Severity**: MEDIUM  
**CWE**: CWE-312 (Cleartext Storage of Sensitive Information)  
**File**: `backend/src/lib/otp-store.ts`

**Description**:  
OTP values are stored as plaintext in Redis: `{ otp: "483920", expiresAt: ..., firstName: ..., lastName: ... }`. If Redis is compromised (no-auth mode, key leakage), attacker can read all active OTPs and user names.

**Fix**:
- Hash OTP before storage: `sha256(otp + identifier + salt)`
- On verify: hash the submitted OTP the same way and compare
- Or: use HMAC for time-based validation

---

### BUG-012: No Request Size Limit
**Severity**: MEDIUM  
**File**: All API routes

**Description**:  
No `Content-Length` limit is configured. A malicious client can send arbitrarily large request bodies, consuming memory.

**Fix**:
- Configure Medusa's body parser limit: `{ limit: '10kb' }`

---

### BUG-013: Checkout Amount Validation — No Upper Bound
**Severity**: MEDIUM  
**File**: `backend/src/api/store/checkout/create-order/route.ts`

**Description**:  
```typescript
if (!amount_in_rupees || amount_in_rupees <= 0) {
  return MedusaResponse.json({ message: "..." }, { status: 400 })
}
// No upper bound check
```
A malformed request with `amount_in_rupees: 999999999999` will pass to Razorpay, which may cause unexpected behavior or expose Razorpay account limits.

**Fix**:
- Derive amount from cart total on the server, not from client input
- Never trust client-provided amounts for payment processing

---

### BUG-014: No Input Validation Schema
**Severity**: MEDIUM  
**File**: All API routes

**Description**:  
All routes use manual checks (`if (!identifier)`) without schema validation. Missing fields are silently undefined, type coercion is unchecked, and unexpected fields pass through.

**Fix**:
- Install `zod`
- Define request schemas for each route
- Validate at entry point; return 422 with field-level errors on failure

---

### BUG-015: Redis Connection Errors Silently Swallowed
**Severity**: MEDIUM  
**File**: `backend/src/lib/redis.ts`

**Description**:  
```typescript
redisClient.on("error", (err) => console.error("Redis error:", err))
```
Redis connection errors are logged but do not cause the application to fail fast or degrade gracefully. OTP operations will throw uncaught errors that propagate as unhandled 500s.

**Fix**:
- Implement circuit-breaker or health-degraded flag on Redis failure
- Surface Redis health in `/health` endpoint
- Alert on sustained Redis errors

---

### BUG-016: No CSRF Protection
**Severity**: MEDIUM  
**CWE**: CWE-352 (Cross-Site Request Forgery)  
**File**: All state-mutating endpoints

**Description**:  
POST endpoints modify state (create orders, authenticate users) without CSRF tokens. Relies entirely on CORS and SameSite cookies.

**Fix**:
- Implement Double Submit Cookie pattern or synchronizer token pattern
- At minimum, validate `Origin` header against allowed CORS origins
